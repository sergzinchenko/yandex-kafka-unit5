## Getting Started
Тестовое задание Яндекс-Практикум по теме "Модуль 5. Безопасность в Kafka"
Результат содержит настройки серверного и клиентского Kafka в среде Docker Compose 
## Folder Structure
    docker-compose.yml                       - Файл для развертывания связанного набора docker-контейнеров 
                                               (kafka - кластер из 3-х брокеров,zookeeper,kafka-ui)
    .env                                     - Переменные среды, используемые скриптах
    kafka_server_jaas.conf                   - Конфигурационный файл JAAS (Java Authentication and Authorization
                                                Service) для каждого брокера Kafka, содержит учётные данные для SASL-аутентификации
    zookeeper.sasl.jaas.conf                 - Конфигурационный файл JAAS для ZooKeeper, содержит учётные данные 
                                               для SASL-аутентификации между ZooKeeper и Kafka брокерами, и между брокерами Kafka и ZooKeeper
    admin.properties                         - конфигурационный фвйл для использования в консольных клиентских
                                               утилитах Kafka с полномочиями адмниистратора
    1_init_server.sh                         - Скрипт bash для автоматичского конфигурирования и запуска кластера 
                                               Kafka и Kafka UI согласно заданию
    2_init_app.sh                            - Скрипт bash для автоматичского конфигурирования прикладных 
                                               артефактов задания (топики и ACL)
    *.keystore.jks                           - Java Key Store-хранилище приватных ключей и публичных сертификатов 
                                               для каждого брокера                            
    *.truststore.jks                         - Java Key Store-хранилище доверенных сертификатов и CA для каждого брокера
    *_keystore_creds                         - Пароль для кранилищ keystore для каждого брокера
    *_sslkey_creds                           - Пароль приватных ключей для каждого брокера
    *_truststore_creds                       - Пароль доверенного хранилища для каждого брокера
    client.truststore.jks                    - Хранилище деверенных сретификатов для Kafka UI

    - 'security': Конфигурация SSL Kafka (используется и заполняется скриптом 1_init_server.sh)
                  Далее для каждого kafka-1, kafka-2, kafka-3 наборы файлов
        *.cnf                                - Кофигурация SSL для каждого брокера кластера (формируется вручную) 
        *.key                                - Приватный ключ SSL-сертификата для каждого брокера
        *.csr                                - Запрос SSL-сертификата для кажлого брокера
        *.crt                                - Сертификаты x509 для каждого брокера, подписанные подписанный CA	
                                               openssl (создаются утилитой openssl)
        *.p12                                - PKCS12-хранилище сертификатов для каждого брокера
        *.keystore.jks                       - Java Key Store-хранилище приватных ключей и публичных сертификатов 
                                               для каждого брокера (на основе *.p12)
        *.truststore.jks                     - Java Key Store-хранилище доверенных сертификатов и CA
        - 'ca': Самоподписанный корневой сертификат
            ca.cnf                           - Конфигурация, на основе которого выпускается корневой сертификат
            ca.key                           - Приватный ключ корневого сертификата
            ca.crt                           - Публичный корневой сертификат
            ca.pem                           - Объединенный контейнер для key и crt (в формате Base64)  
            ca.srl                           - Содержит серийный номер корневого сертификата (создается при генерации)
    - 'app': Скрипты и конфигурации для тестирования Системных и прикладных настроек Kafka (согласно заданию)
        .env                                 - Переменные окружения (SERVER)
        producer.properties                  - Конфигурация консольного продьюсера
        producer_topic-1.sh                  - bash - скрипт для отправки сообщений в topic-1
        producer_topic-2.sh                  - bash - скрипт для отправки сообщений в topic-2
        consumer.properties                  - Конфигурация консольного консьюмера
        consumer_topic-1.sh                  - bash - скрипт для приёма сообщений из topic-1
        consumer_topic-2.sh                  - bash - скрипт для приёма сообщений из topic-2

       
## Инструкции по развертыванию и тестированию
1. На рабочей станции требуется Linux c развернутыми docker и docker compose с доступом в интернет  
2. Запустить скрипт 1_init_server.sh. Выполнится перегенерация файлов в папке security, а также в корневой папке проекта будут пересозданы файлы для настройки используемых docker-образов Kafka (*_creds, *.jks)
3. Запустить скрипт 2_init_app.sh для создания топиков (topic-1 и topic-2) и разграничения доступов к ним (producer - пользователь для отправки сообщений, consumer - пользователь для приёма сообщений).   
4. Выполнить раздельный/паралелльный (для наглядности) запуск прикладных сриптов:
    producer_topic-1.sh (Запись сообщений в topic-1)
    producer_topic-2.sh (Запись сообщений в topic-2)
    consumer_topic-1.sh (Чтение сообщений из topic-1)
    consumer_topic-2.sh (Чтение сообщений из topic-2. Должно закончится неудачно, согласно заданию, ACL на чтение для пользователя consumer отсуствуют)

5. Формат сообщений в топиках:
    - Ключ: topic-n-item_$i
        где n - номер топика (1 или 2)
        i - номер записи (1,2 и т.д.)
        Пример: topic-1-item_3
    - Значение: {"id": $i, "timestamp": "yyyyMMddhhmmss"}
        Пример
            {"id": "3", "timestamp": "20260612210816"}
}
